Api Met Service Layer En In-Memory Data

Doel

Het doel van deze oefening is niet je conceptueel begrip van het de technische onderwerpen uit te diepen, wel de praktische kant in te oefenen.

De bedoeling is dat je dergelijke code stilaan uit het hoofd kan schrijven, zonder (al te veel) opzoekwerk. Gebruik eventueel eens een IDE of editor die je nog niet goed kent.

Start met een nieuwe template en schrijf from scratch een api met service laag en in-memory data voor het beheer van Pokemon.

Api

De api beheert videogames en houdt bij welke pokemon erin voorkomen.

  • CRUD /pokemons

  • CRUD /videogames

Model

Als je moeite hebt om klassendiagrammen te lezen, consulteer dan het relevante hoofdstuk van Softwareontwerp 2.

Rules

Als je daar zin in hebt, kan je het verhaal natuurlijk uitbreiden om verder te oefenen.

  • Een Pokemon die gebruikt wordt in een Videogame kan niet gewist worden.

  • Het wissen van een Videogame zal geen Pokemon wissen.

  • De Naam van een Pokemon is uniek.

  • De Naam van een Videogame is uniek.

  • Geldige waarden voor Soort zijn [Water, Gras, Elektriciteit].

  • Geldige waarden voor Zeldzaamheid zijn [Laag, Normaal, Hoog].

Technisch

  • Mogelijks stoot je op ontwerp problemen. Overweeg de technieken uit de les, maar vertrouw op jezelf en make it work.

  • Als je besluit Enums te gebruiken, zal je mogelijk nood hebben aan onderstaande snippet.
