Opdracht 1 - Wishlist Met Dapper

Foreign Key

Een FK Customer ⇒ Wishlist is niet zelfde als Wishlist ⇒ Customer.

  • Wil je de optie open laten gemakkelijk meerdere lijsten per klant te hebben?

    • Wishlist ⇒ Customer

  • Hoe verhinderen we dat meerdere klanten aan dezelfde wishlist gekoppeld worden?

    • Unique Keys/Indexes

  • Hoe gemakkelijk kunnen we controleren of klant een wishlist heeft?

    • Gemakkelijker wanneer Customer ⇒ Wishlist

Maak een bewuste keuze en weet ze te verdedigen.

Modelbinding Annotations

Gebruik ze. De validatie is grofweg gratis en wordt afgehandeld nog voor jouw api controller code aangeroepen wordt.

Dapper Insert List

Dapper accepteert in param ook een lijst van objecten. Je hoeft dus zelf geen loop te schrijven om meerdere WishlistProducts in één keer te inserten.

SQL Merge

SQL biedt ondersteuning voor een conditioneel statement dat in een enkele dataset verschillende rijen anders (insert/update/delete) kan afhandelen.

Dit is veel performanter dan steeds heen en weer gaan tussen C# en database (al dan niet in een lus).

De basis case is ofwel inserten, ofwel updaten als de sleutelwaarde al bestaat. Dit is een Upsert (Update + Insert).

Moet je plots alles als merge operatie schrijven? Neen. Zal je professioneel occasioneel situaties kunnen herkennen waar dit duidelijk de betere keuze is? Ja.

Volwaardige Merge

  • MS SQL Server (Full Merge)

  • Oracle (Full Merge)

  • Postgres (Full Merge)

Upsert

  • MySQL (Upsert)

  • Postgres (Upsert)

Filter In DB Wanneer Mogelijk

Wanneer je slechts een enkel resultaat (wishlist) zoekt, moet je er voor zorgen dat de filtering is SQL gebeurt. Het is niet efficient om een volledige tabel in te lezen, deze te mappen en dan te filteren in service/api layer.

Dapper Map Met Dictionary

Wanneer je een n-n of 1-n query resultaat wil mappen, moet je een map() functie meegeven aan Dapper. Deze functie zal uitgevoerd worden voor elk record in de result set om uiteindelijk het resultaat van de Query() methode op te bouwen.

Daarna zal je de resultaten nog moeten groeperen / flatmappen (SelectMany).

Je kan er voor opteren dit alles in een keer te doen door buiten de map een Dictionary te declareren en deze in de map te onderhouden.

Deze aanpak heeft een aantal neveneffecten

  • map is geen zuivere functie meer - er zijn side effects. Je verwacht dit niet van een lambda.

  • Het resultaat van de Query() call wordt niet gebruikt.

Hoewel werkend, is het duidelijk niet de elegantste oplossing.

CreatedAt

De 201 met Location header en aangemaakte resource in body is toch wel handig, hé. Gebruik het.

BadRequest Leugen

Een generieke Exception catchen in api controller methode en dan een BadRequest teruggeven is geen goeie manier van werken.

  • Top level generieke catches laat je beter afhandelen door ErrorController.

  • BadRequest antwoorden op generieke Exception is niet oprecht. Hoe weet jij dat het de fout van de Client is als je zelfs niet weet wat er verkeerd liep?

Opdracht 2 - Zet Over Naar EF Core Migrations

Entity Annotations

Gebruik annotations om EF Core en de database informatie te verschaffen over jouw intenties. Een project met EF Core Migrations die geen enkele annotation op de entities heeft, is verdacht.

DbContext Licht Houden

Wanneer je SQL First werkt, wordt er een uitgebreide DbContext gegenereerd.

Wanneer je Code First werkt, hou je de DbContext zo minimaal mogelijk. Nagenoeg alles wat in een SQL First DBContext staat, wordt automatisch toegepast in een Code First scenario op basis van een set conventies.

Voeg enkel toe wat niet vanzelf werkt en je niet gemakkelijk kwijt kan in entity annotations.
